# ChibiForge Component Initialisation — Specification v1

## 1. Purpose

This document specifies the **component initialisation mechanism** for ChibiForge. It covers:

- The `<init>` element added to `component/schema.xml` for declaring an initialisation function.
- The ordering model for init function invocation.
- The `componentsInit()` generated function and how it is produced.
- The impact on the platform component and `main.c`.

---

## 2. Problem Statement

Many components require a one-time initialisation call before they can be used — `halInit()`, `osalInit()`, `lwipInit()`, `vfsInit()`, etc. Without a formal mechanism, the user must manually maintain a sequence of calls in `main.c`, updating it every time a component is added or removed. This is error-prone and couples `main.c` to the component set in a way that undermines the declarative nature of ChibiForge configurations.

The solution is to make initialisation **declarative in the schema** and **generated by the platform component**: each component that requires an init call declares it in its `schema.xml`; the platform component's template reads the full component set at generation time and emits a single `componentsInit()` function that calls all declared init functions in the correct order. `main.c` calls only `componentsInit()` and is otherwise unaware of which components are present.

---

## 3. The `<init>` Element in `schema.xml`

### 3.1 Syntax

The `<init>` element is an optional child of the root `<component>` element, placed alongside `<sections>`:

```xml
<component
    xmlns="http://chibiforge/schema/component"
    id="org.chibios.chibiforge.components.hal.core"
    name="HAL Core"
    version="21.11.0"
    hidden="false"
    is_platform="false">

  <description>...</description>
  <resources/>
  <categories>...</categories>
  <requires>...</requires>
  <provides>...</provides>

  <init function="halInit" order="early"/>

  <sections>
    ...
  </sections>

</component>
```

### 3.2 Attributes

`<init>` has two required attributes:

| Attribute | Type | Description |
|---|---|---|
| `function` | string | Name of the C function to call. Must be a valid C identifier. The function is assumed to have the signature `void <function>(void)`. |
| `order` | enum | Initialisation phase. One of: `first`, `early`, `dontcare`, `late`, `last`. |

### 3.3 `order` values

| Value | Meaning |
|---|---|
| `first` | Must run before all other init calls. Exclusive — at most one component in a configuration may declare `order="first"`. |
| `early` | Runs in the early phase, after `first`. Multiple components may declare `early`; relative order among them is unspecified. |
| `dontcare` | No ordering requirement. Runs after all `early` components and before all `late` components. Relative order among `dontcare` components is unspecified. |
| `late` | Runs in the late phase, after all `dontcare` components. Multiple components may declare `late`; relative order among them is unspecified. |
| `last` | Must run after all other init calls. Exclusive — at most one component in a configuration may declare `order="last"`. |

The full invocation sequence is therefore:

```
first  →  early (unordered among peers)  →  dontcare (unordered)  →  late (unordered among peers)  →  last
```

### 3.4 Exclusivity of `first` and `last`

The generator MUST enforce that at most one component in a configuration declares `order="first"` and at most one declares `order="last"`. If multiple components declare the same exclusive order, the generator emits an error (not a warning) and aborts generation. This is a hard error because the generated `componentsInit()` would be ambiguous and potentially incorrect.

### 3.5 Components with no `<init>`

A component that declares no `<init>` element requires no initialisation call. The generator silently skips it. This is the common case for purely static or header-only components.

### 3.6 Multiple init functions per component

A component may declare at most one `<init>` element. If a component requires multiple initialisation steps at different phases, it must consolidate them into a single wrapper function that it owns, and declare that wrapper in `<init>`.

## 4. The `componentsInit()` Generated Function

### 4.1 Ownership and location

The `componentsInit()` function is generated by the **platform component**. It is delivered via the platform's `cfg/` template directory and lands in:

```
generated/<normalised_platform_id>/components_init.c
```

This file is always overwritten on generation (`cfg/` semantics). It is a pure generated artefact — the user never edits it.

### 4.2 Template responsibility

The platform component's `components_init.c.ftlc` template iterates over all components in the active configuration, collects those that declare an `<init>` element, sorts them by phase order, and emits forward declarations and the `componentsInit()` body.

The template has access to the full configuration data model, which includes every component's schema metadata including their `<init>` declarations.

### 4.3 Generated file structure

Example output for a configuration containing platform, kernel/RT, OSAL, HAL Core, lwIP, and VFS:

```c
/*
 * componentsInit() — generated by ChibiForge, do not edit.
 * Platform: org.chibios.chibiforge.components.platform.stm32g4xx
 * Generated: <timestamp>
 */

/* Forward declarations */
void osalInit(void);
void halInit(void);
void lwipInit(void);
void vfsInit(void);

void componentsInit(void) {

  /* Phase: early */
  osalInit();
  halInit();

  /* Phase: dontcare */
  lwipInit();

  /* Phase: late */
  vfsInit();
}
```

Within each phase, the order of calls among peers follows the order in which components appear in `chibiforge.xcfg` (configuration order). This is deterministic and user-controllable by reordering components in the configuration file.

### 4.4 Header declaration

The platform component also generates a companion header:

```
generated/<normalised_platform_id>/components_init.h
```

Contents:

```c
/*
 * components_init.h — generated by ChibiForge, do not edit.
 */

#ifndef COMPONENTS_INIT_H
#define COMPONENTS_INIT_H

#ifdef __cplusplus
extern "C" {
#endif

void componentsInit(void);

#ifdef __cplusplus
}
#endif

#endif /* COMPONENTS_INIT_H */
```

This header is included by `main.c` (generated once, write-once by the platform component).

---

## 5. Impact on `main.c`

The platform-generated `main.c` scaffold (write-once, `cfg_root_wo/`) includes `components_init.h` and calls `componentsInit()` as the first action:

```c
#include "components_init.h"

int main(void) {
  componentsInit();

  /* User code starts here */

  for (;;) {
  }
}
```

Because `main.c` is write-once, the user may freely extend it after initial generation. The `componentsInit()` call is the stable contract — it never needs to change regardless of which components are added to or removed from the configuration, because the generated `components_init.c` absorbs all such changes.

## 6. Init Order Assignments for Foundation Components

The following table records the `order` declared by each foundation component that requires initialisation:

| Component | Function | `order` |
|---|---|---|
| `org.chibios.chibiforge.components.hal.osal` | `osalInit()` | `first` |
| `org.chibios.chibiforge.components.hal.core` | `halInit()` | `early` |
| `org.chibios.chibiforge.components.oslib` | `chLibInit()` | `early` |
| `org.chibios.chibiforge.components.kernel.rt` | `chSysInit()` | `early` |
| `org.chibios.chibiforge.components.kernel.nil` | `chSysInit()` | `early` |

Notes:

- `osalInit()` is `first` because it initialises the OS abstraction layer that all subsequent init functions may rely on for synchronisation primitives.
- `halInit()` is `early` because it depends on the OSAL being initialised first.
- `chSysInit()` is `early` because it starts the scheduler; in typical ChibiOS usage it is called after `halInit()`. Relative ordering between `halInit()` and `chSysInit()` within the `early` phase is determined by component order in `chibiforge.xcfg`.
- Platform and chport components do not declare `<init>` — their initialisation is part of the startup sequence (`boardInit()`, `SystemInit()`) which runs before `main()`.
- Board component does not declare `<init>` — `boardInit()` is called from the platform startup, not from `componentsInit()`.

## 7. Init Order Assignments for Planned Components

Anticipated `order` values for planned component classes:

| Component class | Anticipated function | Anticipated `order` |
|---|---|---|
| lwIP | `lwipInit()` | `dontcare` |
| VFS Core | `vfsInit()` | `late` |
| LittleFS (standalone) | `lfsInit()` | `dontcare` |
| FatFS (standalone) | `fatfsInit()` | `dontcare` |
| SB Core | `sbInit()` | `late` |
| USB class drivers | driver-specific | `late` |
| Crypto | `cryptoInit()` | `dontcare` |

These are indicative and will be confirmed when each component class is specified.

## 8. Schema XSD Impact

The `<init>` element must be added to `chibiforge_schema.xsd`. The addition to the `componentType` complex type:

```xml
<element name="init" type="tns:initType" maxOccurs="1" minOccurs="0"/>
```

New type definition:

```xml
<complexType name="initType">
  <attribute name="function" use="required">
    <simpleType>
      <restriction base="string">
        <minLength value="1"/>
        <pattern value="[a-zA-Z_][a-zA-Z0-9_]*"/>
      </restriction>
    </simpleType>
  </attribute>
  <attribute name="order" use="required">
    <simpleType>
      <restriction base="string">
        <enumeration value="first"/>
        <enumeration value="early"/>
        <enumeration value="dontcare"/>
        <enumeration value="late"/>
        <enumeration value="last"/>
      </restriction>
    </simpleType>
  </attribute>
</complexType>
```

## 9. Generator Engine Impact

The generator engine must be extended to:

1. **Parse `<init>`** from each component's `schema.xml` when loading the component registry.
2. **Validate exclusivity** of `first` and `last` across all components in a configuration before generation. Emit a hard error if violated.
3. **Expose init data to templates** — the FMPP data model must include, for each component in the configuration, the declared `function` and `order` values (or null if no `<init>` is declared). The platform template relies on this to generate `components_init.c`.
4. **Sort by phase** in the template or in the engine data model. Phase sort order is: `first=0`, `early=1`, `dontcare=2`, `late=3`, `last=4`.

---

## 10. Open Items

1. **`chSysInit()` ordering relative to `halInit()`**: in standard ChibiOS usage `chSysInit()` is typically called before `halInit()`. The exact relative order within the `early` phase should be confirmed against ChibiOS documentation and locked down in the foundation component specs.

2. **Conditional init**: some components may only need initialisation under certain configuration conditions (e.g. a driver component where the user has disabled all driver instances). A future `condition` attribute on `<init>` could suppress the call based on a `@cond:` expression. Deferred to v2.

3. **Init return values**: the current spec assumes `void <function>(void)`. If any ChibiOS init function returns an error code, the signature assumption breaks. All known ChibiOS init functions are void; this is noted for verification when each component is authored.
